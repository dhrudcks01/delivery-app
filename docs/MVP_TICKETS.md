# 서비스 MVP 핵심 흐름 (요약)
1) USER: 쓰레기 수거 신청 생성(주소/연락처/요청사항)
2) OPS_ADMIN: 기사 배정(ASSIGNED)
3) DRIVER: 현장 무게 입력 + 사진 업로드(MEASURED)
4) 서버: 요금 계산(무게 기반) → 저장된 카드로 자동 결제 시도
5) 결제 성공: PAID → COMPLETED
6) 결제 실패: PAYMENT_FAILED(운영자 확인/재시도/결제수단 재등록)

### [x] T-0406 수거완료 이후 결제 처리 정책 고도화(자동결제 + 결제대기풀 일괄처리) 및 사용자 노출 상태 단순화
**배경**
- 현재 상태 타임라인에 `결제 대기`, `결제 완료`가 노출되어 사용자 관점에서 단계가 과도하게 보인다.
- 운영 관점에서는 자동결제 실패/지연 건을 한 번에 처리할 수 있는 결제 대기풀 관리가 필요하다.

**Goal**
- 기사 수거완료(MEASURED) 이후 결제 실행 정책을 명확히 분리한다.
  - 기본: 저장카드 자동결제 즉시 시도
  - 보조: 결제 대기풀(PAYMENT_PENDING) 적재 후 운영자 일괄 결제 실행
- 사용자(USER) 노출 프로세스는 `수거완료 -> 처리완료` 중심으로 단순화한다(결제 중간 상태 미노출).

**DoD**
- 결제 실행 정책:
  - MEASURED 전이 시 자동결제 시도(기존 정책 유지)
  - 자동결제 미실행/재시도 대상은 `PAYMENT_PENDING`으로 적재 가능하도록 정책 확정(조건 문서화)
  - OPS_ADMIN용 `PAYMENT_PENDING` 목록 조회 + 다건(일괄) 결제 실행 API 추가
- 상태 모델/노출 분리:
  - 내부 상태(`PAYMENT_PENDING`, `PAID`)는 운영/감사용으로 유지 가능
  - USER API/모바일 타임라인에서는 결제 중간 상태를 숨기고 `MEASURED(수거완료)` 이후 성공 시 `COMPLETED(처리완료)`만 노출
  - OPS_ADMIN/SYS_ADMIN 화면은 내부 상세 상태를 그대로 확인 가능
- 회귀/정합성:
  - 결제 성공 시 최종 상태는 `COMPLETED`로 일관
  - 결제 실패/대기 건은 USER에게 일반화된 안내 문구로 노출하고, 운영 화면에서만 상세 사유 확인
  - 상태 매핑 정책 문서 및 OpenAPI 응답 예시 업데이트
- 테스트:
  - 자동결제 성공: `MEASURED -> COMPLETED` USER 노출 검증
  - 자동결제 실패 후 대기풀 적재: USER/OPS 상태 노출 차이 검증
  - 대기풀 일괄결제 성공/부분실패 시나리오 검증(멱등 포함)
